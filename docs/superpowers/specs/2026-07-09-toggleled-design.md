# toggleled — blink main-CPU GPIO 25 over the FwGUI link

**Date:** 2026-07-09
**Status:** Approved

## Purpose

A minimal wilibsp app for the FreeWili 2 **display CPU** (RP2350B) that
toggles **main-CPU GPIO 25** every 500 ms. It does this by sending OneWili
commands over the FwGUI display link — UART0 at 8,000,000 baud with hardware
flow control (Rx=GPIO0, Tx=GPIO1, RTS=GPIO2, CTS=GPIO3). The main CPU must be
running the default FreeWili 2 firmware, which carries the OneWili display
bridge.

This is also the first integration of the generated OneWili C API into the
wilibsp tree, establishing `libs/` as the home for reusable non-BSP libraries.

## Components

### `libs/onewili/` — OneWili C API for WiliBSP (new, copied)

A verbatim copy of the generated library from
`freewili-firmware/freewilimain/testprojects/onewili/wilibsp`
(source of truth — regenerated by the menutool API Generator; if the API is
regenerated, re-copy the directory):

- `src/` — `onewili.c`, `onewili_events.c`, `binary_framing.c`,
  `binary_transport.c`, `onewili_fwgui.c`
- `include/` — `onewili.h`, `onewili_enums.h`, `onewili_events.h`,
  `onewili_binary.h`, `onewili_binary_framing.h`, `onewili_fwgui.h`
- `CMakeLists.txt` — builds the `onewili_fwgui` STATIC library; links
  `pico_stdlib hardware_uart hardware_gpio`; `include/` is PUBLIC.
- `README.md`, `examples/blink.c` — kept as documentation.

A comment noting the source-of-truth path is added to the copied
`CMakeLists.txt`. No other modifications — the copy stays diffable against
the generator output.

### `apps/toggleled/` — the app (new, scaffolded)

Scaffolded with `fw new-app toggleled`, then:

- `CMakeLists.txt`: link `freewili2_bsp` **and** `onewili_fwgui`; keep
  `pico_set_binary_type(toggleled copy_to_ram)` (repo invariant — all code
  runs from SRAM).
- `main.c`:
  1. `board_init()` — **must run before** `ow_open_fwgui()`: it raises
     clk_sys to 250 MHz and re-sources `clk_peri`, and `uart_init` derives
     the baud divisor from `clk_peri` at call time. (8 Mbaud needs
     baud×16 ≤ clk_peri; 128 MHz ≤ 250 MHz — fine.)
  2. `ow_open_fwgui(&dev)` — if it fails (handshake timeout, e.g. main CPU
     not running the bridge), `DIAG(...)` the failure over RTT, sleep 1 s,
     retry forever. No silent exit.
  3. Loop: `ow_io_gpio_set_io_toggle(&dev, 25); sleep_ms(500);`
- `README.md`: one paragraph — what it does, that the main CPU must run
  stock firmware, how to build/flash.

### Top-level `CMakeLists.txt` (edited)

Add `add_subdirectory(libs/onewili)` (before the apps) and
`add_subdirectory(apps/toggleled)` (in the existing apps block).

## Error handling

- `ow_open_fwgui` failure: RTT diagnostic + 1 s retry loop (see above).
- Command-call failures in the blink loop (`ow_io_gpio_set_io_toggle`
  returning non-`OW_OK`, e.g. link dropped mid-run): `DIAG` the status and
  continue looping — the link recovers via flow control and the handshake
  already happened; do not re-open.

## Verification

- `fw build toggleled` compiles with no warnings from the new code.
- On hardware (user-performed): `fw flash toggleled`, main CPU running stock
  FreeWili 2 firmware → main-CPU GPIO 25 toggles at 1 Hz; `fw rtt` shows the
  open-failure diagnostic when the link is absent.

## Out of scope

- Binary/text event polling (the lib supports it; this app sends only).
- Any display/LED output on the display CPU itself.
- Automated host tests — the lib is generated code and the app is I/O glue.
