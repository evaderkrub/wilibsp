# On-hardware findings — CC1101 sub-GHz radio (Increment 3)

**Date:** 2026-07-04
**Board:** FreeWili2, RP2350B rev 3, on a CMSIS-DAP probe (VID:PID `0x2e8a:0x000c`).
**App:** `apps/hello_cc1101` (RTT-only self-verify: probe → RSSI sweep → same-pad TX→capture).
**Result:** ✅ **PASS** — all three phases, after fixing one integration bug found on hardware.

## What ran

`fw flash hello_cc1101` (OpenOCD, program+verify+reset) then `fw rtt`. Final RTT:

```
ioexp: init ok (LCD reset released; CC1101 + 433 MHz antenna default)
=== hello_cc1101: sub-GHz radio smoke test ===
cc1101: PARTNUM=0x00 VERSION=0x14
cc1101: probe PASS
scan: row=ok floor=-103 dBm  peak=-87 dBm @ 433015625 Hz
gdo_capture: init pio2 sm=0 dma=0
capture: sent=24 pulses, drained=24 edges (nonzero => PIO2/DMA path live)
cc1101: smoke test complete
```

Interpreting each line against the on-hardware watch list from the whole-branch review:

1. **Probe** — `PARTNUM=0x00 VERSION=0x14`: a real CC1101 answered over SPI1. PARTNUM 0x00 is correct for the part; VERSION 0x14 is the documented TI value and is neither 0x00 nor 0xFF, so the presence check passed. This confirms the `spi_bus` arbiter's GPIO8 mux (LCD_DC ↔ CC1101 MISO) and stale-RX-FIFO drain recover real MISO data, not LCD garbage.
2. **RSSI sweep** — `floor=-103 dBm, peak=-87 dBm @ 433015625 Hz`: a sane noise floor and an in-band peak. Confirms per-hop `SRX` + the fixed 400 µs AGC/RSSI settle read settled RSSI (not an AGC transient) with per-hop auto-cal off and a single band-center `SCAL`.
3. **Capture** — `pio2 sm=0 dma=0`, `sent=24 pulses, drained=24 edges`: the full chain (pio2 SM → ENDLESS DMA ring → `write_addr`-polled `dma_head()` → drain) is live, and `pio_set_gpio_base(pio2, 16)` really does reach GDO0 = GPIO32. Notably the drained count was exactly 24 — the review's M2 concern (a floating Hi-Z GDO0 adding spurious edges before `ook_tx_send` drives it) did **not** materialize on this board; the count matched the pulses sent.

## Bug found and fixed: SPI1 was never initialized for radio-only apps

**Symptom.** The first flash+RTT hung: output stopped right after the banner, before `cc1101_init()`'s own `PARTNUM/VERSION` DIAG — i.e. the hang was *inside* `cc1101_init()`, before its first diagnostic.

**Root cause (systematic-debugging).** The SPI1 SSP enable (`spi_init`) plus routing SCK/GPIO10 and MOSI/GPIO11 to `GPIO_FUNC_SPI` existed **only** in `st7796_init()` (`bsp/display/st7796.c:87-89`). `board_init()` brought up the shared bus's *other* facets (CC1101 CS parked high, I²C1, I/O expander) but not the SSP itself. The `spi_bus` arbiter (`spi_bus_acquire_cc1101`) only re-baudrates and drains the RX FIFO — it assumes the SSP is already enabled. `hello_cc1101` never calls `st7796_init()`, so SPI1 stayed disabled and `spi_write_read_blocking` spun forever on a peripheral whose TX FIFO never signals writable.

This was latent because every prior app (`hello_display`, `hello_audio`) initializes the display, which incidentally brought up SPI1. Increment 3 is the first SPI1 consumer that does not use the display.

**Fix** (`fix(platform): bring up shared SPI1 in board_init for radio-only apps`). The shared SPI1 bus is a platform resource, so its bring-up belongs in the platform, not the display driver. Added `spi_bus_init()` to `bsp/platform/spi_bus.c` (enable the SSP at `LCD_SPI_BAUD`, route SCK/MOSI to `GPIO_FUNC_SPI` at 12 mA — mirroring the display's proven setup) and called it from `board_init()` immediately after the `clk_peri` re-source. `st7796_init()` is left unchanged; its `spi_init` is now an idempotent re-init to the same config (nothing is mid-transfer between the two calls), so the proven display path is byte-for-byte unchanged in behavior.

**Regression check.** After the fix, `hello_display` still boots and streams live `touch` events over RTT (display SPI + touch I²C + main loop all healthy), and `fw test` stays 8/8. The clock-config invariant (vreg → 250 MHz → clk_peri re-source) is untouched — the fix only adds one `spi_bus_init()` call after it.

## Notes for future increments

- **Any non-display app that uses SPI1** (radio, and later NFC/other) now works out of the box because `board_init()` brings up the shared bus. Do not rely on `st7796_init()` for SPI1 bring-up.
- The `pio2` capture is independent of audio (pio0) and LEDs (pio1); this run exercised only the radio, so cross-PIO coexistence (base-16 shift on pio2 not perturbing pio0 I²S / pio1 WS2812) remains architecturally sound but was not co-exercised on silicon here. Worth one combined run if a future app drives audio/LEDs and the radio together.
- The demo's phase-3 is a same-pad plumbing test, not an over-the-air RX test (GDO0 is one pin: chip RX-out XOR MCU TX-in). Verifying a real RX demod path needs an external 433 MHz transmitter — a separate bench test, out of scope for this increment.
